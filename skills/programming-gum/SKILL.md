# Programming Gum

## Overview

Gum is a UI framework for MonoGame that uses XML to define user interface elements. While Gum Tool provides a WYSIWYG editor, all components are stored as XML files that can be created and edited manually. This skill provides XML format reference and patterns for programmatic UI creation.

## When to Use

- Creating or editing Gum UI files (.gumx, .gusx, .gucx, .gutx, .behx)
- Building MonoGame UIs without using Gum Tool
- Debugging Gum XML structure errors
- Understanding Gum component patterns
- Creating game inventory systems, HUDs, menus
- Working with sprite sheets and texture atlases in UI

## ⚠️ CRITICAL: File Organization

**Gum projects REQUIRE specific directory structure. Files in wrong locations will not load!**

```
GumProject/
├── GumProject.gumx          # Main project file
├── Components/              # ALL .gucx files MUST go here
│   ├── YourComponent.gucx
│   └── Controls/            # Subdirectories are part of component name
│       └── Button.gucx      # Referenced as "Controls/Button"
├── Screens/                 # ALL .gusx files MUST go here
│   └── MainScreen.gusx
├── Standards/               # Standard element definitions (.gutx)
└── Behaviors/               # Behavior definitions (.behx)
```

**Key Rules:**
- Component names **include the subdirectory path**: `Controls/ButtonStandard`, not `ButtonStandard`
- Files placed in project root will **not be found**
- The project file references must match the file system structure exactly

## Component vs Screen Decision

**Use Component (.gucx) when:**
- Creating reusable UI elements
- Building widgets, buttons, panels, controls
- Element will be instantiated multiple times
- Element can be used in other components or screens
- Examples: Button, ItemSlot, HealthBar, InventoryPanel

**Use Screen (.gusx) when:**
- Creating top-level application screens
- Building main menu, game HUD, settings screen
- Single-instance full-screen UIs
- Root containers that aren't reused elsewhere
- Examples: MainMenu, GameplayScreen, PauseMenu

**Most custom UI widgets should be Components, not Screens!**

## File Types

| Extension | Type | Purpose |
|-----------|------|---------|
| `.gumx` | GumProjectSave | Main project file referencing all elements |
| `.gusx` | ScreenSave | Top-level UI containers (screens) |
| `.gucx` | ComponentSave | Reusable UI components |
| `.gutx` | StandardElementSave | Primitive element definitions |
| `.behx` | BehaviorSave | Interface/requirement definitions |

## Generated Code vs Custom Code

Each screen and component generates **two** C# files:

| File | Purpose | Safe to Edit? |
|------|---------|---------------|
| `ComponentName.cs` | Custom code file - your event handlers, logic, properties | ✅ **Yes** |
| `ComponentName.Generated.cs` | Auto-generated by Gum - visual structure, bindings | ❌ **No - will be overwritten** |

**Important:** Put ALL your custom code in the `.cs` file (not `.Generated.cs`). Use the `partial void CustomInitialize()` method for setup.

See [common-pitfalls.md#-editing-generatedcs-files](./references/common-pitfalls.md#-editing-generatedcs-files) for details.

## Common Patterns

### Basic Component Structure
```xml
<?xml version="1.0" encoding="utf-8"?>
<ComponentSave xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xmlns:xsd="http://www.w3.org/2001/XMLSchema">
  <Name>ComponentName</Name>
  <BaseType>Container</BaseType>
  <State>
    <Name>Default</Name>
    <!-- Variables here -->
  </State>
  <Instance>
    <Name>ChildName</Name>
    <BaseType>ElementType</BaseType>
    <DefinedByBase>false</DefinedByBase>
  </Instance>
  <Behaviors />
</ComponentSave>
```

### Variable Pattern
```xml
<Variable>
  <Type>TYPE</Type>
  <Name>PropertyName</Name>
  <Value xsi:type="xsd:TYPE">VALUE</Value>
  <SetsValue>true</SetsValue>
</Variable>
```

### Common Layouts
- **Fill parent (preferred)**: WidthUnits=1, Width=100 (PercentageOfParent at 100%)
- **Fill parent (alternative)**: WidthUnits=2, Width=0 (RelativeToParent with 0 offset)
- **Size to children**: WidthUnits=4 (RelativeToChildren)
- **Center element**: XUnits=6, XOrigin=1, YUnits=7, YOrigin=1
- **Vertical stack**: Use `Controls/StackPanel` or ChildrenLayout=1 (TopToBottomStack)
- **Horizontal stack**: Use `Controls/StackPanel` or ChildrenLayout=2 (LeftToRightStack)
- **Auto-grid**: ChildrenLayout=3, AutoGridHorizontalCells=N, AutoGridVerticalCells=M

## Common Mistakes

### Missing xsi:type Attribute
**Problem:**
```xml
<Value>128</Value>  <!-- Missing xsi:type -->
```

**Solution:**
```xml
<Value xsi:type="xsd:float">128</Value>
```

The `xsi:type` must match the variable Type (string→xsd:string, float→xsd:float, int→xsd:int, bool→xsd:boolean).

### Not Validating XML Before Testing
**Problem:** A malformed XML tag (e.g., `<Type>DimensionUnitType</Name>`) causes Gum Tool to fail with a generic "Could not deserialize the XML file" error that's hard to debug.

**Solution:** Always run XMLLint on .gucx files before testing in Gum Tool:
```bash
xmllint --noout path/to/Component.gucx && echo "XML is valid"
```

Add this verification step to your workflow for all new or modified .gucx files.

### Wrong DimensionUnitType Values
**Problem:** Using incorrect enum values leads to unexpected sizing behavior.

**Solution:** Reference [element-properties-quick-reference.md](./references/element-properties-quick-reference.md) for correct values:
- 0 = Absolute
- 1 = PercentageOfParent
- 2 = RelativeToParent
- 4 = RelativeToChildren

### Forgetting SetsValue=true
**Problem:**
```xml
<Variable>
  <Type>float</Type>
  <Name>Width</Name>
  <Value xsi:type="xsd:float">128</Value>
  <!-- Missing SetsValue -->
</Variable>
```

**Solution:** Always include `<SetsValue>true</SetsValue>` when setting a variable value.

### Incorrect Parent-Child Relationships
**Problem:** Child not appearing because parent reference is missing or incorrect.

**Solution:**
```xml
<Variable>
  <Type>string</Type>
  <Name>ChildInstance.Parent</Name>
  <Value xsi:type="xsd:string">ParentInstance</Value>
  <SetsValue>true</SetsValue>
</Variable>
```

### States Without Categories
**Problem:** Creating states directly without a category (except Default state).

**Solution:** All non-Default states must be in a Category:
```xml
<Category>
  <Name>ButtonCategory</Name>
  <State>
    <Name>Enabled</Name>
    <!-- Variables here -->
  </State>
</Category>
```

### Not Using ExposedAsName
**Problem:** External code forced to use internal implementation details like `TextInstance.Text`.

**Solution:** Expose clean API:
```xml
<Variable>
  <Type>string</Type>
  <Name>TextInstance.Text</Name>
  <Value xsi:type="xsd:string">Button</Value>
  <ExposedAsName>DisplayText</ExposedAsName>
  <SetsValue>true</SetsValue>
</Variable>
```

Now external code can use `DisplayText` instead.

### Missing Font Configuration (Text Elements)
**Problem:** Text not rendering correctly or using default font.

**Solution:** Always include font configuration with `IsFont=true` marker:
```xml
<Variable>
  <IsFont>true</IsFont>  <!-- REQUIRED for font selection -->
  <Type>string</Type>
  <Name>TextInstance.Font</Name>
  <Value xsi:type="xsd:string">Arial</Value>
  <SetsValue>true</SetsValue>
</Variable>
<Variable>
  <Type>int</Type>
  <Name>TextInstance.FontSize</Name>
  <Value xsi:type="xsd:int">18</Value>
  <SetsValue>true</SetsValue>
</Variable>
<Variable>
  <Type>bool</Type>
  <Name>TextInstance.IsBold</Name>
  <Value xsi:type="xsd:boolean">false</Value>
  <SetsValue>true</SetsValue>
</Variable>
```

### Missing Standard State Variable
**Problem:** Components not working with state management systems.

**Solution:** Every component should include the standard State variable:
```xml
<Variable>
  <Type>State</Type>
  <Name>State</Name>
  <Value xsi:type="xsd:string">Default</Value>
  <Category>States and Visibility</Category>
  <SetsValue>false</SetsValue>
</Variable>
```

### Not Leveraging Existing Components
**Problem:** Creating complex layouts with raw elements when components exist.

**Solution:** Use existing components from Controls/ directory:
```xml
<!-- Instead of Container with ChildrenLayout -->
<Instance>
  <Name>MyStackPanel</Name>
  <BaseType>Controls/StackPanel</BaseType>  <!-- Use existing component -->
  <DefinedByBase>false</DefinedByBase>
</Instance>
```

Common useful components:
- `Controls/StackPanel` - Vertical/horizontal stacking with behavior
- `Controls/ButtonStandard` - Styled button with states
- `Controls/Panel` - Bordered container
- `Controls/ScrollViewer` - Scrollable content area

### PercentageOfParent vs RelativeToParent
**Problem:** Confusion between similar sizing modes.

**Solution:**
- **PercentageOfParent (1)**: Width/Height as percentage (100 = 100% of parent)
- **RelativeToParent (2)**: Width/Height as offset from parent size (0 = same as parent, -20 = 20 pixels smaller)

For "fill parent", prefer PercentageOfParent:
```xml
<Variable>
  <Type>DimensionUnitType</Type>
  <Name>WidthUnits</Name>
  <Value xsi:type="xsd:int">1</Value>  <!-- PercentageOfParent -->
  <SetsValue>true</SetsValue>
</Variable>
<Variable>
  <Type>float</Type>
  <Name>Width</Name>
  <Value xsi:type="xsd:float">100</Value>  <!-- 100% -->
  <SetsValue>true</SetsValue>
</Variable>
```

## Integration with MonoGame

Gum XML files are loaded at runtime using:
- **MonoGameGum** - Runtime library for MonoGame
- **GumRuntime** - Core runtime without platform dependencies

The XML files define the UI structure; runtime code handles rendering, input, and state management.

### Entry Point

Use `GumService.Default` as the main entry point:

```csharp
// Initialize
GumService.Default.Initialize(this, "GumProject/GumProject.gumx");

// Update (in Game.Update)
GumService.Default.Update(gameTime);

// Draw (in Game.Draw)
GumService.Default.Draw();
```

**Note:** Some projects define a convenience property like `private GumService GumUI => GumService.Default;`. This is a project-specific alias, not a Gum framework class.
